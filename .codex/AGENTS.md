# Global Profile — Agent Orchestrator Project

## Персонализация

- Язык общения: **русский**
- **Имею право не соглашаться** с решениями. Если решение ведёт к костылю, дыре в безопасности или техдолгу — ОБЯЗАН возразить и предложить альтернативу. Молчаливое согласие с плохим решением = ошибка.
- **Качество и security > скорость.** Не принимать "потом поправим", "сойдёт для MVP", "это временно". Временные решения становятся постоянными.
- **Долгосрочная польза > быстрый результат.** Выбирать решения, которые масштабируются и поддерживаются.
- Если пользователь настаивает на костыльном решении — чётко обозначить риски и зафиксировать в Report.

## Инварианты стека (нельзя нарушать)

Выбор технологий фиксирован. Предлагать или использовать альтернативы **ЗАПРЕЩЕНО**.

### Backend
- **Язык:** Python 3.11+
- **Фреймворк:** Flask
- **LLM:** OpenAI API (`openai` SDK)
- **Data:** pandas
- **Storage:** SQLite (через `sqlite3` или `SQLAlchemy`)
- **Env:** python-dotenv (`.env` файл для секретов, НИКОГДА не хардкодить API-ключи)

### Frontend
- **Стек:** HTML / CSS / Vanilla JS
- **Никаких фреймворков** (React, Vue, Angular — запрещено)
- **Никаких CDN-зависимостей** без явного согласования

## Общие правила (STRICT — применяются ко всем профилям)

### Структура проекта

```
project/
├── app/
│   ├── __init__.py          # Flask app factory
│   ├── routes/              # Blueprint'ы по доменам
│   ├── agents/              # Логика агентов
│   ├── services/            # Бизнес-логика, OpenAI-клиент
│   ├── models/              # SQLite модели / схемы
│   └── utils/               # Хелперы, pandas-утилиты
├── static/                  # CSS, JS, assets
├── templates/               # Jinja2 HTML-шаблоны
├── swarm-report/            # Отчёты задач (создаётся агентом)
├── tests/                   # pytest
├── .env                     # Секреты (НЕ коммитить)
├── .env.example             # Пример переменных без значений
├── requirements.txt
└── AGENTS.md                # Этот файл
```

### Правила безопасности (OWASP)

- API-ключи и секреты — **только через `.env`**, никогда в коде
- Все SQL-запросы — **только параметризованные**, никакой конкатенации строк
- Входные данные от пользователя — **всегда валидировать** на backend
- OpenAI промпты, содержащие пользовательский ввод — **санитизировать** (prompt injection)
- Ошибки — логировать на сервере, клиенту отдавать **общее сообщение** без деталей стека

### Качество кода

- **Типизация:** аннотации типов везде (`def get_agent(agent_id: int) -> Agent:`)
- **Docstring:** у каждой публичной функции и класса
- **Линтер:** `flake8` / `ruff` — без предупреждений
- **Тесты:** `pytest`, покрытие новой логики обязательно
- **Логирование:** `logging` (не `print`)

### Передача контекста между стадиями

При переходе между стадиями ОБЯЗАТЕЛЬНО передавать:
1. Исходный запрос пользователя
2. Краткий итог предыдущей стадии
3. Если откат — причину отката

## Профиль: Бизнес-фича

### Development Workflow (STRICT)

#### Стадии
1. **Research** — исследование задачи, кодовой базы, зависимостей
2. **Plan** — формирование плана реализации
3. **Executing** — написание кода
4. **Validation** — проверка результата (тесты, сборка, UI)
5. **Report** — отчёт о работе
6. **Done** — задача завершена

#### Разрешённые переходы

```
Research   -> Plan
Research   -> Executing
Plan       -> Executing
Executing  -> Validation
Executing  -> Research
Validation -> Report
Validation -> Executing
Validation -> Research
Report     -> Done
```

Все остальные переходы **ЗАПРЕЩЕНЫ**.

#### Research — Консилиум агентов

Все агенты запускаются **параллельно**, каждый анализирует задачу со своей экспертизы:

| Роль              | Зона ответственности                                      |
|-------------------|-----------------------------------------------------------|
| Архитектор        | Структура модулей, зависимости, паттерны Flask            |
| AI/LLM-эксперт    | Дизайн промптов, цепочки агентов, OpenAI API best practices|
| Безопасность      | OWASP, prompt injection, утечки данных, авторизация       |
| Data-инженер      | Схема SQLite, pandas-пайплайны, миграции                  |
| Frontend-эксперт  | UI/UX, JS-логика, взаимодействие с API                    |

**Порядок:**
1. Запустить всех агентов параллельно с описанием задачи
2. Собрать результаты, сформировать сводное резюме
3. Уточнить через вопрос пользователю, если есть расхождения
4. Только после полного сбора контекста — переход в Plan

#### Validation — порядок работы

**Шаг 1:** Определить, что проверяем:
- Backend API (curl / pytest)
- Frontend (браузер / HTML-проверка)
- Агентная логика (unit-тесты с mock OpenAI)

**Шаг 2:** Создать E2E сценарий:

```
./swarm-report/<slug>-e2e-scenario.md
```

```markdown
# E2E Scenario: <название>
Платформы: Backend, Web

## Шаги
- [ ] 1. Запустить Flask сервер
- [ ] 2. POST /api/agents — создать агента
- [ ] 3. Проверить ответ 201 + поле `agent_id`
- [ ] 4. GET /api/agents/{id} — получить агента
- [ ] 5. Проверить корректность данных в SQLite
```

**Шаг 3:** Запустить тесты:

```bash
# Установка зависимостей
pip install -r requirements.txt

# Линтер
ruff check . || flake8 .

# Тесты
pytest tests/ -v --tb=short
```

**Шаг 4:** Обновить чеклист после каждого шага (`[ ]` → `[x] ✅`).

**Шаг 5:** При ошибках — откат в Executing с описанием проблемы.

#### Report — содержимое отчёта

```
./swarm-report/<slug>-<YYYY-MM-DD>.md
```

Содержимое:
- Название фичи и дата
- Описание задачи
- Итоги Research (сводка консилиума)
- План реализации
- Что реализовано (файлы, функции, endpoints)
- Результаты Validation (тесты, покрытие, curl-примеры)
- Проблемы и откаты (если были)
- Статус: `Done` / `Частично`

## Профиль: Поиск бага

### Bug Hunting Workflow (STRICT)

#### Стадии
1. **Reproduce** — воспроизведение бага
2. **Diagnose** — диагностика корневой причины
3. **Fix** — исправление
4. **Validation** — проверка фикса + регрессии
5. **Report** — отчёт
6. **Done** — завершено

#### Разрешённые переходы

```
Reproduce  -> Diagnose
Reproduce  -> Report         (баг не воспроизводится)
Diagnose   -> Fix
Diagnose   -> Reproduce      (нужно воспроизвести иначе)
Diagnose   -> Report         (фикс не требуется / невозможен)
Fix        -> Validation
Fix        -> Diagnose       (фикс выявил другую причину)
Validation -> Report
Validation -> Fix            (фикс не работает)
Validation -> Diagnose       (причина была другой)
Report     -> Done
```

#### Reproduce — воспроизведение

1. Получить описание бага (traceback, лог, шаги)
2. Попытаться воспроизвести через `curl`, `pytest` или браузер
3. Зафиксировать в файл:

```
./swarm-report/<slug>-reproduce.md
```

```markdown
# Reproduce: <описание бага>
Платформа: Backend / Web
Статус: Воспроизведён / Не воспроизведён

## Входные данные
- Описание: ...
- Traceback/лог: ...

## Шаги воспроизведения
1. curl -X POST /api/agents -d '{"name": "test"}'
2. → Ожидалось: 201, Фактически: 500 Internal Server Error

## Логи / вывод
```
[ERROR] ...
```
```

4. Если не воспроизводится после 3 попыток — уточнить окружение или перейти в Report с пометкой.

**Перед каждой попыткой — перечитать файл `*-reproduce.md`. Не повторять проверенные сценарии.**

#### Diagnose — Консилиум агентов

Все агенты запускаются **параллельно**:

| Роль           | Зона ответственности                                        |
|----------------|-------------------------------------------------------------|
| Диагност       | Анализ traceback, логов, стектрейсов Python                 |
| Архитектор     | Архитектурные причины, зависимости модулей Flask            |
| AI/LLM-эксперт | Ошибки OpenAI API, rate limits, prompt issues               |
| Безопасность   | Уязвимости, утечки, проблемы авторизации                    |

**Порядок:**
1. Передать всем агентам: описание бага + шаги воспроизведения + traceback
2. Собрать гипотезы
3. Сформировать сводный диагноз (root cause + затронутые файлы + способ исправления)
4. Уточнить через вопрос, если гипотезы расходятся

#### Report — содержимое отчёта бага

```
./swarm-report/<slug>-<YYYY-MM-DD>.md
```

Содержимое:
- Название бага и дата
- Описание проблемы
- Шаги воспроизведения (из Reproduce)
- Диагноз (root cause, сводка консилиума)
- Что исправлено (файлы, функции, строки)
- Результаты Validation (тесты, curl-примеры)
- Проблемы и откаты (если были)
- Статус: `Fixed` / `Not Reproducible` / `Partially Fixed` / `Won't Fix`

## Запрещено всегда

- Хардкодить API-ключи, пароли, токены в коде
- Конкатенировать SQL строками (всегда параметризация)
- Использовать `print()` вместо `logging`
- Игнорировать исключения через голый `except:`
- Коммитить `.env` файл
- Переходить между стадиями минуя разрешённые пути
- Использовать технологии вне инвариантов стека
